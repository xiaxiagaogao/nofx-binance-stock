# 链式推理 Agent 架构升级计划

**分支**: `feature/chain-of-thought`  
**状态**: 规划中，未动工  
**优先级**: 高（在现有系统稳定后实施）

---

## 背景

现有系统每个交易周期只发起一次 LLM 调用，要求 agent 在单次推理中同时完成：
宏观对齐 → 技术分析 → 组合审查 → 风险评估 → 最终决策

这导致各环节相互干扰，决策质量存在天花板。

---

## 目标

将单次 LLM 调用拆分为多个串行步骤，每步专注一件事，前一步输出作为下一步输入，提升决策的逻辑严密性和胜率。

---

## 五步推理链设计

### Step 1 — 宏观对齐
**输入**: 当前宏观论文（market_regime / portfolio_intent / sector_bias / key_risks）  
**任务**: 根据宏观环境，判断当前周期适合操作哪些板块，哪些方向不碰  
**输出**: 本周期可操作板块列表 + 禁区列表 + 简短理由

### Step 2 — 技术筛选
**输入**: Step1 可操作板块 + 候选标的行情数据（4H/1H EMA、RSI、OI、资金费率）  
**任务**: 从技术面找出结构清晰、位置良好的标的  
**输出**: 技术面通过的标的列表 + 每个标的的关键支撑/阻力位

### Step 3 — 组合审查
**输入**: Step2 候选列表 + 当前持仓明细  
**任务**: 考虑现有仓位的集中度、板块占比、全局持仓数，筛除真正没有空间的机会  
**输出**: 最终有仓位空间的机会列表

### Step 4 — 决策生成
**输入**: Step3 结论  
**任务**: 对每个机会生成具体决策（方向、仓位、入场价、止损位、止盈目标、置信度）  
**输出**: 结构化交易指令（与现有格式兼容）

### Step 5 — 风险验证
**输入**: Step4 决策列表  
**任务**: 逐条检查是否违反风控规则（杠杆、保证金占比、板块限额、全局上限）  
**输出**: 通过 / 调整 / 拒绝，附原因

---

## 技术实现方案

### 改动范围
- `kernel/engine.go` — 新增 `RunChainOfThought()` 方法，串联5次 LLM 调用
- `trader/auto_trader_loop.go` — 在 `runCycle()` 中替换原有单次调用
- `store/strategy.go` — 新增 `EnableChainOfThought bool` 开关字段，默认 false

### 兼容性原则
- 新增开关字段，默认关闭，不影响现有逻辑
- 每步调用使用独立的 context，避免 token 膨胀
- 调用失败时降级回单次调用，保证系统不中断

### Token 消耗估算
- 现有: ~3,000 tokens/周期
- 升级后: ~8,000–12,000 tokens/周期（5步 × 平均2,000 tokens）
- 关盘时段 2h/次，开盘时段 20min/次，整体成本可控
